# ChalkTalk（チョークトーク）企画書

## プロジェクト概要

**プロジェクト名：ChalkTalk（チョークトーク）**
**コンセプト：** AIが台本から動画まで自動生成する、YouTube解説動画メーカー
**タグライン：** 「黒板の前に立つだけ。台本も、動画も、AIにおまかせ。」

---

## 1. 解決する課題

YouTube解説動画の制作には、台本作成・立ち絵の選定・背景の用意・動画編集と多くの工程が必要で、個人クリエイターにとって大きなハードルとなっている。特に「ゆっくり解説」系の動画は人気が高い一方、制作にはデスクトップ専用ソフトが必要で、初心者には参入障壁が高い。

ChalkTalkは、テーマを入力するだけでAIが台本・演出指示を自動生成し、ブラウザ上で動画を完成させることで、この課題を解決する。

## 3. 主要機能

### Phase 1：MVP（最小実行可能プロダクト）

#### 3.1 AI台本生成
- ユーザーがテーマ（例：「量子コンピュータとは」）を入力
- AIがYAML形式で台本を自動生成
- 各シーンごとにセリフ・背景指定・立ち絵の表情/配置を出力

**YAML出力例：**
```yaml
title: "量子コンピュータ入門"
scenes:
  - id: 1
    background: "blackboard"
    text_on_board: "量子コンピュータとは？"
    characters:
      - name: "sensei"
        position: "right"
        expression: "smile"
    dialogue: "今日は量子コンピュータについて解説していきます！"
    duration: 4

  - id: 2
    background: "blackboard"
    text_on_board: "従来のコンピュータ → 0 or 1\n量子コンピュータ → 0 and 1"
    characters:
      - name: "sensei"
        position: "right"
        expression: "explain"
    dialogue: "従来のコンピュータはビットを使いますが、量子コンピュータは量子ビットを使います。"
    duration: 6
```

#### 3.2 YAMLエディタ
- 生成されたYAMLを手動で編集可能
- リアルタイムプレビュー（シーン単位のスナップショット表示）
- バリデーション付き

#### 3.3 クライアントサイド動画生成
- 事前用意済みの背景画像・立ち絵画像をCanvas API上で合成（レイヤー重ね合わせ）
- テキスト（黒板上の文字・字幕）のみCanvasで描画
- MediaRecorder APIで動画ファイル（WebM）としてエクスポート
- サーバーサイド不要 → コスト0

#### 3.4 動画ダウンロード
- WebM形式で即ダウンロード
- （将来）MP4変換オプション

### Phase 2：拡張機能

| 機能 | 概要 | 優先度 |
|------|------|--------|
| TTS音声合成 | ブラウザのWeb Speech APIまたはVOICEVOX連携 | 高 |
| テンプレート機能 | 「対話形式」「一人語り」等のテンプレ | 高 |
| キャラクターカスタマイズ | 立ち絵の色・アクセサリ変更 | 中 |
| BGM追加 | フリーBGMの選択・挿入 | 中 |
| MP4エクスポート | FFmpeg.wasm によるブラウザ内変換 | 中 |
| 字幕自動生成 | セリフからSRT/VTT出力 | 低 |
| 共有機能 | YAML台本のURL共有 | 低 |

### 4.3 動画レンダリングの仕組み（コア技術）

**方針：Canvas APIは「合成機」として使用。背景・立ち絵はすべて事前制作済みの画像アセット。**

Canvas APIの役割は、用意済みの画像素材を重ね合わせて1枚のフレームを作る「合成」のみ。画像の生成や描画は行わない。

```
YAMLパース → シーンオブジェクト配列を生成
    ↓
各シーンについて：
    1. Canvas に事前用意済みの背景画像を配置
    2. 立ち絵画像を指定位置に重ねて配置（表情差分は画像切替）
    3. 黒板上のテキストをオーバーレイ描画（チョーク風フォント）
    4. 字幕テロップをオーバーレイ描画
    5. MediaRecorder で指定duration分キャプチャ
    ↓
全シーンを結合 → WebM Blob → ダウンロードリンク生成
（Phase 2で FFmpeg.wasm によるMP4変換を追加）
```

**この設計のメリット：**
- サーバーサイド処理が不要 → 動画生成コスト¥0
- ユーザーのブラウザで完結 → スケーラビリティの心配不要
- YouTube はWebMアップロードに対応 → MVP段階ではMP4変換不要

## 5. アセット戦略

**基本方針：背景画像・立ち絵はすべて事前に制作・用意し、R2にホスティングする。**
**制作ツール：Gemini画像生成（コスト最小優先）**

### 5.1 背景画像（事前制作済み）
- 黒板（メイン背景）
- ホワイトボード
- 教室
- シンプル単色背景
- **形式：** PNG（1920×1080）、R2から配信
- **拡張：** ユーザーからの要望に応じて追加

### 5.2 立ち絵（Gemini画像生成 → 後処理）

#### キャラクター構成
- **初期：** アニメ・イラスト調のキャラクター（2体）
  - 先生キャラ（解説役）：指し棒・メガネなど「先生らしい」記号を付与
  - 生徒キャラ（質問・リアクション役）
- **表情差分：** 各キャラ6種（通常・笑顔・驚き・考え中・説明・困り）
- **形式：** 透過PNG（後処理で背景除去）、R2から配信
- **将来：** ユーザーが独自立ち絵をアップロード可能に

#### Gemini画像生成ワークフロー

**Step 1：キャラデザ確定（プロンプトテンプレート方式）**

1体につき1つの「マスタープロンプト」を作成し、キャラの外見を細かく固定する。

```
マスタープロンプト例（先生キャラ）：
「アニメイラスト調の女性キャラクター。全身立ち絵。
 年齢20代後半。身長165cm程度。
 髪：ピンクとシアンのグラデーションボブカット、肩にかかる長さ。
 服装：紺色のブラウス、黒いタイトスカート、白いエプロン。
 小物：右手に指し棒を持っている。
 体型：スリム。ポーズ：正面向き、やや左手を腰に。
 背景：単色グレー（#808080）。
 画風：アニメセル画調、輪郭線あり、影は2段階。」
```

ポイント：
- 背景は「単色グレー」を指定 → 背景除去が容易になる
- 服装・髪型・小物をできるだけ具体的に記述
- 「全身立ち絵」「正面向き」を毎回含める
- 満足いく1枚が出るまで数回〜十数回ガチャを回す

**Step 2：表情差分の生成**

マスタープロンプトの末尾に表情指定を追加して生成する。

```
表情指定の追加部分：
- 通常：「穏やかな無表情、口は閉じている」
- 笑顔：「目を細めて笑っている、口は開いて歯が見える」
- 驚き：「目を大きく見開いている、口がOの字に開いている」
- 考え中：「目を閉じ、人差し指をあごに当てている」
- 説明：「片目を閉じてウインク、人差し指を立てている」
- 困り：「眉を八の字にして、汗のマークが浮かんでいる」
```

注意点：
- 同じプロンプトでもGeminiは毎回異なる画像を生成するため、
  体型・服装・ポーズがブレた場合は再生成（ガチャ）を繰り返す
- 「許容範囲の一貫性」を目標とし、完璧は求めない
- 目安：1キャラ×6表情で、20〜50回程度の生成が必要

**Step 3：後処理パイプライン**

```
Gemini出力（背景付きPNG）
    ↓
① 背景除去：remove.bg（無料枠）またはrembg（ローカル、完全無料）
    ↓
② リサイズ・トリミング：統一サイズに調整（高さ1080px基準）
    ↓
③ 品質チェック：黒板背景に重ねて違和感がないか目視確認
    ↓
④ ファイル命名：sensei_normal.png, sensei_smile.png, ...
    ↓
⑤ R2にアップロード
```

推奨ツール（すべて無料）：
- 背景除去：rembg（Python、ローカル実行、完全無料・無制限）
- リサイズ：ImageMagick または GIMP
- 一括処理：シェルスクリプトで自動化可能

#### 一貫性を保つためのTips
- Geminiの生成結果から「最も出来の良い1枚」をまず選び、その画像を参考画像として以降のプロンプトに添付する
- 体のポーズは固定し、顔の表情だけを変える指示にする
- どうしてもブレる場合：体部分は1枚目を流用し、顔だけGIMPで差し替える（最終手段）
- 将来的にStable Diffusion（inpainting）への移行も選択肢として残しておく

### 5.3 アセット管理
- すべてのアセットはCloudflare R2に格納
- アセット一覧はJSONマニフェストで管理（キャラ名・表情名・ファイルパスの対応表）
- YAMLの `expression: "smile"` → マニフェスト経由で `sensei_smile.png` を解決

```json
{
  "characters": {
    "sensei": {
      "name": "先生",
      "expressions": {
        "normal":   "characters/sensei/sensei_normal.png",
        "smile":    "characters/sensei/sensei_smile.png",
        "surprise": "characters/sensei/sensei_surprise.png",
        "think":    "characters/sensei/sensei_think.png",
        "explain":  "characters/sensei/sensei_explain.png",
        "trouble":  "characters/sensei/sensei_trouble.png"
      }
    },
    "student": {
      "name": "生徒",
      "expressions": { ... }
    }
  },
  "backgrounds": {
    "blackboard": "backgrounds/blackboard.png",
    "whiteboard": "backgrounds/whiteboard.png"
  }
}
```

### 6.2 変動費（AI台本生成）

Claude Haiku利用の場合（1台本あたり約2,000トークン入出力と仮定）:

| 月間台本生成数 | 推定API費用 |
|--------------|------------|
| 1,000本 | ~$1（約¥150） |
| 10,000本 | ~$10（約¥1,500） |
| 100,000本 | ~$100（約¥15,000） |

### 6.3 コスト最適化戦略

- **レート制限：** IPベースで1日あたりの生成回数を制限
- **キャッシュ：** 類似テーマの台本結果をKVでキャッシュ
- **段階的課金：** 無料枠を超えるヘビーユーザーにはAPI Key持ち込み（BYOK）を案内
- **動画生成コスト：** クライアントサイド処理のため¥0

## 7. ユーザー情報の取り扱い

### 7.1 基本方針：最小限の情報取得

| 項目 | 方針 |
|------|------|
| ユーザー登録 | 不要（匿名利用可能） |
| Cookie | 必要最小限（セッション管理不要） |
| 生成データ | ローカル保存（ブラウザのlocalStorageまたはダウンロード） |
| アクセスログ | Cloudflare Analyticsの標準機能のみ |
| 個人情報DB | 持たない |

### 7.2 API Key持ち込み（BYOK）モデル
- ユーザーが自分のClaude API Keyを入力して利用
- Keyはブラウザのローカルにのみ保存（サーバーに送信しない）
- サーバー側のAPI費用を削減しつつ、ユーザーデータを扱わない

### 7.3 将来の認証（必要になった場合）
- Cloudflare Access の利用を検討
- OAuthのみ（パスワード管理を回避）

## 9. リスクと対策

| リスク | 影響度 | 対策 |
|--------|--------|------|
| AI API費用の急増 | 高 | レート制限 + BYOK + キャッシュ |
| ブラウザ互換性（MediaRecorder） | 中 | Chrome/Edge優先、非対応ブラウザは案内表示 |
| 立ち絵の一貫性（Gemini生成のブレ） | 中 | マスタープロンプト固定 + 顔だけ手動差替の最終手段 + 将来SD移行オプション |
| 立ち絵の著作権・利用規約 | 中 | Gemini生成画像の利用規約を遵守、商用利用可否を確認 |
| 生成コンテンツの不適切利用 | 中 | 利用規約で禁止事項を明記 |
| Cloudflare無料枠の制限変更 | 低 | 代替サービス（Vercel等）への移行プラン |
| 動画品質（クライアントサイド制限） | 中 | 解像度オプション、長時間動画の分割処理 |

## 12. まとめ

ChalkTalkは「ブラウザだけで完結する無料のAI解説動画メーカー」として、YouTube解説動画制作の民主化を目指す。Cloudflareのインフラを最大限活用し、クライアントサイド処理を軸にすることで、個人開発でも持続可能な低コスト運営を実現する。

ユーザー情報を持たない設計により、運営リスクを最小化しつつ、「テーマを入れるだけで動画が完成する」という圧倒的な手軽さで、多くのユーザーに届けることを最優先する。
